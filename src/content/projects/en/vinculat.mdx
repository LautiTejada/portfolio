> **Confidentiality note**: Vinculat is an internal application without public access. User data and specific metrics have been omitted. The description accurately reflects the technical work performed.

## Context

Vinculat is an organizational management platform for health and social services entities. It manages three user types (Staff, Patients, and Clients) with very different access needs and functionalities. The app centralizes role-based dashboards, user and membership administration, benefits management, and targeted notification delivery.

I participated as a fullstack developer in a 2-person team, building both the backend (NestJS) and the frontend (Next.js).

## Solution — Backend

A REST API was built with **NestJS 10** and **TypeScript**, connected to **PostgreSQL** via **MikroORM 6**. The backend handles all business logic, authentication, authorization, and data processing.

### Authentication and authorization

- JWT with access token (15 min) + refresh token (7 days) using Passport.js and bcrypt.
- System of **18 hierarchical roles** organized by department (HR, Legal, Administration, Clinical, Workshops, etc.).
- Granular authorization based on **resource + action** with custom guards.
- Assignable role policies defining which roles each department can create.

### Multi-role memberships

- A user can have multiple roles within a membership.
- Each membership generates a **unique sequential ID** per organization with typed format: `S-000123` (Staff), `P-04567` (Patient), `C-00988` (Client).
- Uniqueness validation at the database level.

### Benefits with segmentation

- Benefits with lifecycle: `DRAFT → ACTIVE → ENDED`.
- Configurable time windows and visibility segmented by user type and role.
- Per-user claim limits and M:N categorization.
- Optimized indexes for active benefit queries.

### Image pipeline

- Upload with **Sharp** generating multiple variants (original, 256px, 64px, WebP) for avatars and benefits.
- Immutable cache of 1 year and path traversal protection.

### Statistics dashboard

- Raw SQL queries via **Knex** for metrics on active/inactive users by type, benefits by status, and redemption rates.
- Optional time filters for period-based analysis.

## Solution — Frontend

The frontend was developed with **Next.js 15** (App Router), **React 19**, and **TypeScript**. The interface consumes the backend's REST API and presents differentiated views based on user type and permissions.

### Authentication and protected routes

- Authentication with **NextAuth**, session management and token refresh.
- Automatic redirection to dashboards based on user type and role.

### Role-based navigation

- Role-based access control for multiple internal areas.
- Differentiated views for administration, staff, clients, and patients.

### Management modules

- User, benefits, and segmented notification management.
- Forms with **React Hook Form** + **Zod** for typed validation.
- Dashboards with metrics and charts using **Recharts**.
- Styling with **Tailwind CSS** for visual consistency.

## Technical decisions

- **NestJS**: opinionated framework with dependency injection, decorators, and modular structure. Ideal for complex APIs with multiple domains.
- **MikroORM over TypeORM**: Unit of Work pattern, better migration support, and more flexible query builder.
- **PostgreSQL**: relational model needed for the complexity of relationships between users, memberships, roles, benefits, and claims.
- **Next.js App Router + NextAuth**: framework-integrated authentication, nested layouts for role-based views, and Server Components for efficient loading.
- **Sharp for images**: server-side processing generating multiple resolutions, avoiding heavy image loading on mobile devices.
- **Helmet + custom guards**: HTTP-level security and granular authorization on every endpoint.
