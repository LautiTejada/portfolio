> **Confidentiality note**: this project was developed under a confidentiality agreement. Company names, users, and specific data have been anonymized. The technical description is faithful to the work performed.

## Problem

The internal platform handled permissions with a binary system: admin or user. This forced granting full access to people who only needed to view certain modules, creating security risks and accidental modifications to critical data.

Multiple departments (operations, finance, HR, sales) shared the same platform, each with different access needs. The IT team received constant tickets to "give person X access to section Y," which they resolved manually by editing the database.

## Solution

An RBAC (Role-Based Access Control) system was designed with three levels:

1. **Roles**: logical groupings of permissions (e.g., "Operations — Read Only", "Finance — Admin").
2. **Permissions**: resource + action combinations (e.g., `reports:read`, `users:create`, `invoices:delete`).
3. **Assignment**: each user receives one or more roles. Permissions are resolved by union.

An admin panel was built where the IT team manages roles and assignments without touching the database.

## Technical Decisions

- **RBAC over ABAC**: the role-based model is simpler to understand, maintain, and audit for the organization's size (~200 users). ABAC would add unnecessary complexity.
- **Authorization middleware in Express**: each endpoint validates permissions at the middleware level, preventing business logic from being contaminated with access checks.
- **PostgreSQL with relation tables**: normalized schema with `roles`, `permissions`, `role_permissions`, `user_roles` tables. Permission resolution queries use optimized JOINs with indexes.
- **In-memory permission cache**: user permissions are resolved at login and cached in the JWT token (with 15-minute expiration). This avoids queries on every request.
- **Frontend**: `<Gate permission="reports:read">` components that conditionally render based on the current user's permissions.

## Architecture

Backend:

- `middleware/authorize.ts` — Middleware that extracts permissions from JWT and validates against the endpoint
- `services/permissions.ts` — Permission resolution logic (role union)
- `routes/admin/roles.ts` — Role and assignment CRUD
- `db/migrations/` — Migrations for the permissions schema

Frontend:

- `context/AuthContext.tsx` — Context with user and resolved permissions
- `components/Gate.tsx` — Declarative component for UI access control
- `pages/admin/roles/` — Role management panel

## Results

- Migration of 200+ users to the new system without downtime or access loss.
- "Grant access" tickets reduced by 90%: the IT team uses the panel directly.
- Zero unauthorized access incidents since implementation.
- Access auditing implemented: every action is logged with user, timestamp, resource, and result (allowed/denied).

## What I Would Do Differently

I would use a proven library like CASL for permission evaluation on the frontend instead of a custom Gate — ours works but doesn't cover edge cases like conditional permissions based on resource ownership. On the backend, I would consider Open Policy Agent (OPA) if the system needed to scale to more complex policies.
