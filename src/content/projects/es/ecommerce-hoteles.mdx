## Problema

La cadena hotelera dependía completamente de plataformas de terceros (Booking, Expedia) para recibir reservas online. Esto implicaba comisiones de entre 15% y 25% por reserva, sin control sobre la experiencia del usuario ni acceso directo a los datos del cliente.

Tenían un sitio web institucional estático que mostraba fotos y teléfono de contacto, pero no permitía reservar ni consultar disponibilidad online.

## Solución

Se desarrolló una plataforma de reservas directas con tres componentes principales:

1. **Sitio público**: catálogo de hoteles con búsqueda por fecha y ubicación, páginas de detalle por propiedad, y flujo de reserva con pago online.
2. **Motor de disponibilidad**: servicio que calcula disponibilidad en tiempo real cruzando reservas existentes, bloqueos manuales y reglas de tarifa.
3. **Panel de administración**: interfaz para que el equipo del hotel gestione propiedades, habitaciones, tarifas estacionales, y vea el estado de las reservas.

## Decisiones técnicas

- **Next.js App Router**: SSR para las páginas de detalle de hotel (SEO crítico para captar tráfico orgánico), client components para el buscador y el flujo de checkout.
- **PostgreSQL con esquema relacional**: tablas para propiedades, habitaciones, tarifas (con rangos de fechas), reservas, y bloqueos. Queries de disponibilidad optimizadas con índices en rangos de fechas.
- **Stripe Checkout**: se delegó el flujo de pago completo a Stripe para evitar manejar datos de tarjeta. Webhooks para confirmar pagos y actualizar estado de reserva.
- **Optimistic UI en el checkout**: se reserva un "hold" temporal de 15 minutos al iniciar el pago para evitar doble booking mientras el usuario completa el formulario de Stripe.
- **Tailwind CSS**: diseño responsive mobile-first, crítico porque el 70%+ del tráfico hotelero viene de móviles.

## Arquitectura

```
src/
├── app/
│   ├── (public)/         # Sitio público (hoteles, búsqueda, checkout)
│   ├── admin/            # Panel de administración (protegido)
│   └── api/
│       ├── availability/ # Motor de disponibilidad
│       ├── bookings/     # CRUD de reservas
│       └── webhooks/     # Stripe webhooks
├── lib/
│   ├── availability.ts   # Lógica de cálculo de disponibilidad
│   ├── stripe.ts         # Cliente y helpers de Stripe
│   └── db/               # Prisma client y queries
└── components/
    ├── search/           # Buscador y filtros
    ├── booking/          # Flujo de reserva
    └── admin/            # Componentes del panel
```

El deploy está en Vercel (frontend + API routes) con PostgreSQL en un servicio managed. Stripe maneja todo el procesamiento de pagos.

## Resultados

- La plataforma procesa reservas directas eliminando la comisión de intermediarios.
- El motor de disponibilidad responde en < 200ms para búsquedas complejas (múltiples propiedades, rango de fechas de hasta 30 días).
- El panel admin permite al equipo del hotel gestionar tarifas estacionales sin intervención técnica.
- Mobile-first: el flujo completo de búsqueda → detalle → reserva → pago funciona sin fricción en móvil.

## Qué haría diferente

Implementaría un sistema de caché para las búsquedas de disponibilidad más frecuentes (mismas fechas, misma ciudad) usando Redis, ya que el cálculo en PostgreSQL escala bien pero tiene un costo innecesario para queries repetidas. También separaría el panel admin en un proyecto independiente en lugar de tenerlo como rutas protegidas dentro del mismo Next.js — el bundle del admin no debería afectar al sitio público.
